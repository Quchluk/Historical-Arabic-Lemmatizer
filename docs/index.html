How It Works

Three steps.
One hard problem.

Arabic's root-and-pattern system means a single three-letter root can produce dozens of grammatically valid lemmas with completely different meanings. Classical tools guess. ArLem checks.

1

Candidate Generation

The corpus is tokenized and passed through CAMeL Tools' morphological analyzer, which generates all structurally valid lemma candidates for each surface form. This gives the baseline set of guesses that the next two steps then check.

CAMeL Tools Morphological Parsing Tokenization
2

Vector Projection & Anomaly Detection

Tokens sharing the same assigned lemma are embedded and compared using cosine similarity. Tokens that score unusually low against the rest of their group are flagged — they're likely homographs or root-sharing words that got pulled into the wrong lemma bucket.

3

LLM Correction

Flagged tokens are sent to an LLM via OpenRouter with the surrounding sentence as context. The model re-evaluates which lemma actually fits. This is the expensive step — it only runs on the suspects, not the whole corpus.

Honest Assessment

What it does.
What it doesn't.

This is a research tool in active development, not a solved product. Here is a clear-eyed view of its current capabilities and known limitations.

Working Well
Catches what baselines miss
~40% of the deep semantic errors that standard classical tools silently pass through are flagged and corrected.
Hybrid approach outperforms either tool alone
Classical morphology is fast and transparent. LLMs are context-aware. Neither is sufficient alone — combining them is the point.
Hallucination guard via embedding check
The LLM's output is verified against the semantic map, preventing confident-sounding invented lemmas from entering the corpus.
Efficient — LLM only where needed
TF filtering and cosine distance thresholds mean only genuine suspects reach the expensive LLM step.
Known Limitations
Anisotropy in embedding space
High-dimensional vectors warp direction. Cosine similarity can be misleading — "close" doesn't always mean semantically similar in 768 dimensions.
Root gravity problem
Arabic's root system causes كَتَبَ, كِتَاب, and مَكْتَبَة to cluster tightly, making it hard to detect grammatical shifts within the same root family.
No universal accuracy — corpus matters
Tuned for 8th-century poetry can break on 15th-century legal texts. The OpenITI corpus spans 1,400 years — "universal accuracy" is a moving target.
Ground truth is sparse
Zipf's law applies — 1,000 examples for common forms, one for rare ones. Existing gold standards are built for Modern Standard Arabic, not historical texts.
→
Next: Beyond cosine distance
Moving from simple cosine similarity to trained neural clustering (HDBSCAN / UMAP) to better separate true semantic shifts from topic-level noise in the embedding space.

Deployment

Up in four
commands.

Python 3.8+ required. Set your OPENROUTER_API_KEY environment variable before running the correction module.

bash · Historical-Arabic-Lemmatizer
# 1. Clone the repository
git clone https://github.com/Quchluk/Historical-Arabic-Lemmatizer.git
cd Historical-Arabic-Lemmatizer
# 2. Create virtual environment
python -m venv .venv && source .venv/bin/activate
make install
# 3. Fetch CAMeL morphological models
python -m camel_tools.data_downloader light
# 4a. Interactive web interface
make run
# 4b. Batch pipeline (large corpora)
python -m lemmatizer.processing.batch_processor --input_dir ./data/raw
# Output
Initializing models... [OK]
Processing corpus... flagging suspects via cosine threshold
✓ Anomalies identified and dispatched to LLM resolver.
